# AMCP — implementation evidence and open gaps

This document is an evidence-based status snapshot, not a substitute for the
product Definition of Done. A checked capability has code and automated
verification in this repository; it does not replace acceptance tests on real
hosts.

| Area | Current evidence | Remaining verification or implementation |
| --- | --- | --- |
| Local Agent and Controller | Separate Rust processes, authenticated JSONL protocol, local Unix socket, retry and health flow. Both sides require the same protocol-envelope version before accepting a response or request; Controller also rejects an Agent whose binary major/minor release line differs, while accepting patch releases. The local watcher emits source-change events with a canonical watch root and bounded relative paths. A provider collection failure with an available cache returns the last safe collection plus a stable content-free diagnostic event. The Agent also serves authenticated, bounded `SearchLocal` results from this redacted cache for offline/local operation; `amcp-controller local-search` exposes it without a Controller catalog. It validates scope and never opens native provider files while searching. On reconnect, replayed batches are persisted before the cursor advances; their discovered/inserted counts are reported by `run-once` and recorded as bounded `replayed` diagnostics, independently of the subsequent cache-aware live collection. | Run lifecycle/reconnect acceptance test on a real user account. |
| Central catalog and search | SQLite catalog, source observations, evidence, cursors, private named saved searches, Controller-owned artifact tags, durable content-free collection/search metrics and rebuildable FTS projection. A saved search deliberately stores its query and the same shared scope/filter contract only when a user asks to save it; ordinary search telemetry persists only scope, correlation ID, latency, result count and limit (never the query, previews or result IDs), with 1,000-record retention. Artifact tags are private catalog relations and never modify provider-native files. The Controller creates a private consistent SQLite snapshot before a required schema migration and exposes explicit user-requested snapshots through CLI/Desktop; neither path accesses provider-native files. Each Controller collection persists host/provider scope, request/correlation IDs when available, latency, discovered/inserted/replayed counts and a stable failure kind. `amcp-controller diagnostics --json`, Desktop and MCP diagnostics expose the bounded metrics. `amcp-controller benchmark-search --json` measures p50/p95/max of the production FTS path without outputting query/content/IDs and can assert a requested p95 target. On 2026-07-18, the isolated real five-root Codex smoke catalog (44 artifacts) reported p50/p95 under 1 ms and maximum 1 ms against a 300 ms asserted target; this is a baseline, not representative-scale evidence. MCP provides a structured `result_status`: `not_indexed` for an empty catalog, `not_found` for a no-match indexed search, and safe error classifications for unsupported, untrusted and permission-denied policy outcomes. Diagnostics also calculate stale-source ratio, FTS coverage and central database/WAL/SHM size without measuring native provider files. On Unix, a newly created catalog directory is private to the current user; the catalog, RAG projection and present WAL/SHM sidecars share the same restricted path policy, and symlink catalog paths are rejected. One shared search contract supports host, provider, project/trust state, artifact type, lifecycle, maximum sensitivity and observed-date constraints for the desktop and MCP. Newly received source-change events mark matching catalog artifacts stale; a subsequent same-hash collection restores them to active. `scripts/acceptance-recovery.sh` verifies Controller restart idempotence, a consistent explicit catalog backup, bounded index rebuild and population of a fresh central catalog from an Agent fixture; `scripts/acceptance-shared-search.sh` verifies matching source-linked evidence through Controller search and MCP. | Measure search latency against representative user data. |
| Codex provider | Redacted discovery, bounded session metadata/read, app-server runtime read and approved archive/unarchive. The provider descriptor reports detected Codex home, explicitly configured SQLite home, CLI availability and a bounded CLI version without opening native SQLite. A selected discovered session JSONL source can be read as a bounded, redacted, read-only evidence excerpt; it cannot be proposed for mutation. User configuration and root-level `*.config.toml` profiles are inventoried as distinct configuration layers. Recognized rules, skills, hooks and MCP configuration are classified as instruction or tooling without execution. Existing session `cwd` metadata discovers a validated project root (preferring a Git root) and assigns the session to it without reading project content. Desktop users can add up to 32 existing absolute project roots for local collection; those paths are deduplicated, passed only to the local Agent and do not grant mutation trust. A local 2026-07-18 smoke collection against the installed Codex home registered 1 host, 1 provider, 44 artifacts and five normalized real project roots in an isolated Agent run: `agenticswarm`, `azure-mcp`, `CodeRAG`, `gemini-cli`, and session-discovered `sdlc-orchestrator`. The Controller reported healthy provider compatibility, completed incremental FTS projection and zero stale artifacts. | Re-run this smoke after material Codex state/CLI format upgrades. |
| Human UI and embedded Codex | Tauri/React shell, host/provider/project-scoped search, scoped project context for memory/configuration/guidance, session filters for provider/project/branch/model/date/archive state, inspector, change diff/hash/evidence review, human approvals, confirmed rollback, embedded Codex/MCP, redacted app-server event streaming, cancellable turns, content-free diagnostics, and a host/provider-scoped audit trail. The desktop has an explicit evidence-grounded recommendation workflow: a user selects 2–4 redacted catalog results, the local no-network app-server prompt requires citations for factual claims, and the response retains a visible list of the selected source-linked artifacts. If the embedded Codex app-server is unavailable, the UI explicitly indicates degraded mode while catalog search and inspection remain read-only; a later turn retries the runtime. Diagnostics surface bounded stale-source, project-trust, edit-conflict and provider-diagnostic metadata without artifact or transcript bodies. Embedded Codex turns use an explicit no-network read-only sandbox; AMCP changes remain proposal-only until separately approved by a human. The macOS bundle embeds target-built Agent, Controller and MCP sidecars, resolving them without a user `PATH` dependency. `scripts/acceptance-desktop-bundle.sh` verifies all four executables, bundle ID/icon and an ad-hoc resource-sealed signature; `scripts/release-macos.sh` provides explicit hardened-runtime signing, Apple notarization and Gatekeeper verification when release credentials are supplied. | Run desktop smoke test on a Developer ID-signed and notarized application bundle. |
| Safe mutation | Change sets, one-time signed approvals, expected hashes, backup, conflict detection, verification and rollback. Diagnostics expose aggregate applied/conflicted/rolled-back counts without operation content. Codex Agent allowlists the same registered project roots it discovers, but permits mutation only when a project is explicitly `trusted`; untrusted or unknown project files are rejected at proposal and apply. `config.toml` and root-level Codex `*.config.toml` profiles are parsed before proposal and again before any write, preventing invalid TOML from entering a change set or partially applying a multi-operation set; profiles elsewhere remain rejected. The desktop exposes approve/apply only for a proposal and a separately confirmed rollback only for an applied change. Controller audit records cover writes and successful live reads or catalog search results classified `Sensitive` or `SecretLike`, storing identifiers and outcome only, never queries, previews or returned content. `scripts/acceptance-safe-change.sh` verifies Controller → Agent proposal, apply with backup, approved rollback, profile mutation and refusal after an external source edit; `scripts/acceptance-sensitive-read-audit.sh` verifies redaction and matching Controller/MCP search and read audit records in isolated fixture copies. | Exercise each permitted v1 artifact class against non-fixture files. |
| Multi-host | TLS Agent endpoint, pairing/enrollment, host-scoped collection, replay and desktop enrollment/resync. The desktop retains searchable indexed previews for disconnected hosts while disabling live reads and Agent-backed mutation/runtime actions. A private Controller alias can label an enrolled host without changing its Agent-owned `host_id`, native hostname, credentials, scopes or local policy. Users can also create an explicit symmetric relationship between two indexed artifacts from distinct hosts; it is Controller-only metadata and cannot copy, merge or mutate native provider sources. `scripts/acceptance-multi-host-tls.sh` verifies two isolated TLS Agents, a shared Controller catalog and two persisted host connections with a generated CA. | Enroll and reconnect a second physical macOS or Linux host. |
| RAG | Disabled-by-default lexical/vector path, egress consent, source/model invalidation, retention, derived-data clear and redacted evaluation fixture. The Desktop persists a private, validated policy for enablement, allowed host/provider/project scopes, lexical/local-hash/OpenAI-compatible embedding selection, model, retention, chunk size and retrieval budget; MCP consumes the same policy, while egress consent and credentials remain process-local. Ingestion is permanently redacted-excerpt-only and transcript bodies are excluded. Retrieval history is content-free: scope, provider/model metadata, result/citation counters, latency and correlation ID only; user queries/context are never persisted, legacy phrases are scrubbed by migration, and history is capped at 1,000 runs. `amcp-controller rag-evaluate --assert-targets --json` evaluates a redacted corpus through the production lexical retrieval path with configurable citation/recall/forbidden-hit thresholds; it neither enables regular RAG, reads native provider files, persists history nor performs egress. RAG status/Desktop expose average retrieval latency and citation coverage. The desktop can explicitly forget one central memory record: its matching AMCP artifact, FTS projection, RAG chunks and recorded retrieval runs are removed, while a source-hash tombstone prevents a replay from restoring that same native version. Native provider state is not changed. | Establish a representative evaluation corpus and acceptance thresholds for latency, citation coverage and stale-memory rate. |
| Future providers | Claude Code, Kiro and Antigravity adapters support inventory and bounded redacted live reads without changing UI/storage contracts. Provider records persist provider/adapter versions, authorized native roots, a provider-family schema fingerprint, explicit support level, health and Controller compatibility per host/provider. A failed collection served from cache is marked `degraded` without storing its parser error. Controller default collection iterates every registered provider and continues after an individual failure; `scripts/acceptance-multi-provider-collection.sh` verifies fixture-backed Codex, Claude Code, Kiro and Antigravity collection in one run. MCP uses one structured response envelope with request ID, scope, evidence and warnings across providers. | Implement native runtime/session connectors and provider-specific mutation policies only after their public formats/protocols are verified. |
| Linux readiness | XDG controller/Agent state paths, opt-in restricted file credential store, Unix socket, and systemd user-service scripts. | Run systemd installation, enrollment and reconnect acceptance tests on Linux. |
| Windows readiness | Local Controller ↔ Agent uses a Windows named pipe; per-user Scheduled Task install/uninstall scripts provide lifecycle; enrolled credentials use Windows Credential Manager; platform paths and shared-crate CI cover Windows compilation targets. Windows CI also builds/tests and lints the Agent and Controller against the native MSVC environment. | Run Windows-specific filesystem, remote-enrollment and lifecycle acceptance tests on a real host. Local cross-compilation additionally requires a MinGW/MSVC C toolchain. |

## Verification commands

```bash
cargo test --workspace
cargo clippy --workspace --all-targets --all-features -- -D warnings

cd apps/amcp-desktop
cargo test --manifest-path src-tauri/Cargo.toml
cargo clippy --manifest-path src-tauri/Cargo.toml --all-targets -- -D warnings
npm run build
```

The shared crate matrix in `.github/workflows/shared-crates.yml` runs tests and
Clippy on macOS, Linux and Windows. The Windows MSVC job additionally tests and
lints the native named-pipe Agent and Controller; the MCP binary remains
covered by the portable workspace matrix. A separate macOS CI job installs the
desktop dependencies, tests/lints the Tauri backend and runs the signed-resource
desktop bundle acceptance check with AMCP sidecars.
